## C.P5 — Design: Architecture Description

### Cloud Network Design

The Wholesale CRM is deployed on AWS using a **three-tier network architecture** across **two Availability Zones** in eu-west-1 (Ireland):

```
Internet
    │
    ▼
Internet Gateway (igw)
    │
    ▼
┌─────────────────────────────────────────────┐
│  VPC  10.0.0.0/16                           │
│                                             │
│  Public Subnets (10.0.0.0/24, 10.0.1.0/24) │
│  ┌──────────────┐   ┌──────────────────┐   │
│  │ ALB          │   │ NAT Gateway      │   │
│  │ (port 80)    │   │ (outbound only)  │   │
│  └──────┬───────┘   └──────────────────┘   │
│         │ (SG: tasks only)                 │
│  Private Subnets (10.0.10.0/24, 10.0.11.0) │
│  ┌──────▼─────────────────────────────┐    │
│  │ ECS Fargate Tasks (port 3000)      │    │
│  │ min 2 · max 6 · CPU auto-scaling   │────┼──► MongoDB Atlas
│  └────────────────────────────────────┘    │    (via NAT GW)
└─────────────────────────────────────────────┘
```

### Terraform files
| File | Purpose |
|---|---|
| `main.tf` | VPC, subnets, IGW, NAT Gateway, route tables |
| `security.tf` | ALB SG (0.0.0.0/0:80/443) + ECS SG (ALB only:3000) |
| `alb.tf` | Application Load Balancer, target group, HTTP listener |
| `ecs.tf` | ECS cluster, Fargate task definition, service |
| `autoscaling.tf` | CPU target-tracking, min 2 / max 6 tasks |
| `ecr.tf` | Elastic Container Registry for Docker image |
| `ssm.tf` | SecureString parameters (MONGODB_URI, JWT_SECRET) |
| `iam.tf` | ECS execution role (ECR pull + SSM read + CloudWatch logs) |
| `route53.tf` | Private hosted zone + A-alias record pointing at ALB |
| `outputs.tf` | ALB URL, NAT IP, ECR URL |

### Security design decisions
- Tasks run in **private subnets** — no direct internet exposure
- ALB security group allows **80/443 from 0.0.0.0/0** (public web traffic)
- ECS security group allows **port 3000 from ALB SG only**
- Secrets stored in **SSM Parameter Store (SecureString)** — never in code or environment files
- Container runs as **non-root user** (`appuser` in Dockerfile)

## C.D2 — Justify: Is the design effective?

The measured results demonstrate that the architecture satisfies its goals:

**Performance:** The Application Load Balancer distributes traffic across all healthy tasks (evidenced by rotating `X-Instance-ID` headers), keeping p50 latency low even under the heavy test. The target group health-check (`/health`) ensures unhealthy tasks are drained immediately.

**Scalability:** The CPU target-tracking policy (50% threshold) triggered automatic scale-out during the heavy load test, increasing the task count from 2 to N within ~90 seconds of the CPU threshold being breached. This confirms the auto-scaling mechanism works without manual intervention.

**Security:** The private-subnet placement of ECS tasks means they are unreachable directly from the internet; all inbound traffic must pass through the ALB and its security group. Secrets are stored in SSM Parameter Store (SecureString), never in source code or environment files, satisfying the principle of least privilege.

**Resilience:** Multi-AZ deployment across eu-west-1a and eu-west-1b means the service survives a single AZ failure. The ALB automatically stops routing to tasks in a failed AZ and continues serving requests from the healthy AZ.

**Trade-offs:** A single NAT Gateway (not one per AZ) was chosen to reduce cost for this assignment. In production, a NAT Gateway per AZ would eliminate the cross-AZ data transfer cost and remove the single point of failure for outbound traffic.
